Plan: Multi-tenant (SaaS), roles configurables, equipo y carpetas compartidas
Context
AKX Cloud hoy es mono-usuario: cada Usuario tiene un rol string (user/admin, sin uso real) y todo (archivos, carpetas, facturas, fragmentos RAG) se aísla por propietario. El objetivo es venderla como SaaS a varias empresas sobre una sola instancia. Esto exige:

Multi-tenant: una entidad Empresa; todo se aísla además por empresa.
Onboarding de venta: un superadmin (el dueño de la plataforma = Vlad) da de alta empresas y crea el admin de cada una desde un panel propio. Sin auto-registro.
Admin de empresa: gestiona su equipo, sus roles configurables y sus carpetas compartidas. Un usuario tiene 0..N roles funcionales.
Equipo (pestaña solo-admin): crear/editar usuarios y roles, y ver los archivos de cada empleado tal cual los ve él.
Carpetas compartidas por rol: el admin define carpetas y qué roles acceden; los miembros con ese rol las ven en Mis archivos (toggle Personales / Compartido) y los cambios son visibles para todos los que comparten el rol (almacenamiento único, no copias).
Decisiones tomadas con Vlad: instancia compartida multi-tenant · alta de empresas por panel de superadmin · roles funcionales configurables por el admin · UI de archivos con toggle Personales / Compartido · entrega completa por fases (3 fases independientes y testeables).

Tras cada fase: cd backend && npm run dev y cd frontend && npm start (dev loop del CLAUDE.md). Las migraciones TypeORM corren al arrancar la API.

Modelo de datos (objetivo final)
Niveles de cuenta en Usuario.rol (enum): superadmin | admin | miembro (migrar user→miembro). Los roles funcionales (contabilidad, etc.) son entidad aparte y N:N, NO van en Usuario.rol.

empresas (nueva): id, nombre, estado (activa|suspendida), creadoEn.
usuarios (cambia): + empresa ManyToOne (nullable solo para superadmin); rol enum ampliado; N:N con Rol (join usuario_roles).
roles (nueva, por empresa, configurable): id, empresa, nombre (unique por empresa), capacidades (jsonb/text[] de un vocabulario fijo que entiende la app: facturas, busqueda, gestion_archivos, chat…), creadoEn; N:N con Usuario. El admin nombra/asigna roles libremente, pero las capacidades que un rol puede otorgar son ese set fijo (porque el código debe saber qué desbloquea cada una).
carpetas_compartidas (nueva): id, empresa, ruta, nombre, creadoEn; N:N con Rol (join carpeta_compartida_roles) = qué roles acceden.
archivos/carpetas/facturas/fragmentos (cambian): + empresa (NOT NULL para datos de empresa) e índice. archivos/carpetas/fragmentos además + carpetaCompartida ManyToOne nullable: si es null → personal (gobernado por propietario); si tiene valor → compartido (gobernado por empresa + roles, propietario solo audita quién subió).
Ajustar unicidad de Carpeta: personal (propietario, ruta), compartida (carpetaCompartida, ruta).
JWT: añadir empresaId al payload (cambia poco). El rol de cuenta va en el JWT; los roles funcionales se consultan en BD por request (evita permisos rancios al editar roles, dado que el token vive 7d). Actualizar generarToken, JwtPayload, express.d.ts (req.usuario + empresaId).

FASE 3 — Carpetas compartidas por rol + RAG + toggle en Mis archivos
Backend

Entidad CarpetaCompartida + join carpeta_compartida_roles; migración que añade carpetaCompartidaId (nullable) a archivos, carpetas, fragmentos.
Gestión por el admin (en Equipo o sección propia): CRUD de carpetas compartidas + asignar qué roles acceden (/api/equipo/compartido o /api/compartido, soloAdmin).
Helper carpetasCompartidasAccesibles(usuarioId) → ids de carpetas cuyos roles intersecan los roles del usuario (consulta BD). Úsalo para autorizar lectura/escritura.
archivos.service.ts/carpetas.service.ts: soportar ambito = compartido:
listar carpetas compartidas accesibles (raíces) y navegar/subir/editar dentro (poner carpetaCompartidaId, sin filtro por propietario, scoped a empresa + rol).
Subcarpetas libres dentro de una carpeta compartida (como en personal).
rag.service.ts: al indexar un archivo compartido, escribir fragmentos con carpetaCompartidaId; buscarSemantica = fragmentos personales (propietario=yo) UNION fragmentos compartidos accesibles. El worker (tareas.service.ts) deriva el ámbito del propio archivo.
Escaneo/OCR único en compartido: un archivo en carpeta compartida se OCR/escanea e indexa una sola vez al subirse (el resultado vive en BD: textoExtraido, fragmentos, factura). El resto de usuarios del rol reutilizan esa info; nada se reprocesa por usuario. Apóyate en la deduplicación por hashSha256 ya existente y en el almacenamiento único.
Chat consciente del rol (chat.service.ts) — RBAC enforzado en código, nunca en el prompt (los modelos pequeños no son frontera de seguridad fiable):
Mapa tool → capacidad en código: cada tool declara qué capacidad requiere (o ninguna = siempre disponible). Capacidades = grupos: facturas (escanear/listar/ ventas_top/totales/clientes_top/obtener_factura), busqueda (buscar_semantica/ leer_archivo), gestion_archivos (crear/copiar/mover/renombrar/eliminar/papelera), chat (usar el chatbot). Se guardan como array de strings en Rol.capacidades, así partir un grupo en piezas más finas luego NO requiere migración.
Defensa en profundidad: gating en las DOS capas — se cortocircuitan los pre-flights por regex de capacidades ausentes y se filtra el registro de tools del bucle según capacidadesDe(usuario). Si el rol no tiene la capacidad, la tool ni se carga (el dato no es recuperable) y el bot responde "esto no está disponible para tu rol".
System prompt consciente del rol = capa blanda de UX (enfoca al bot en el dominio del rol para que ni lo intente); NO es la frontera de seguridad.
Scope por datos: las tools de búsqueda/lectura operan sobre lo que el usuario puede ver (personal + carpetas compartidas accesibles), reusando los helpers de Fase 3.

core/archivos.service.ts: parámetro ambito / endpoints de compartido.

Limitaciones asumidas en este alcance (revisables luego)
El chat IA respeta capacidades de rol (RBAC enforzado en código, mapa tool→capacidad) y el scope de datos accesibles (personal + compartido del rol). Granularidad por grupos; refinar un grupo en piezas más finas luego no requiere migración (es un array de strings).
Las facturas en carpetas compartidas se escanean una sola vez y son visibles para los usuarios del rol, pero no se atribuyen a un único propietario (la analítica personal de facturas sigue centrada en lo personal; ampliar la analítica compartida queda para después).
Self-serve de empresas (registro público + trial/billing) queda fuera; el modelo de datos lo permite encima del panel de superadmin.
Los roles funcionales se consultan en BD por request (cambios de rol surten efecto sin re-login); solo rol de cuenta y empresaId viajan en el JWT.
